# 通用偏好

- 用中文回复，代码注释用中文，注释写 why 不写 how
- 简洁直接，不要多余总结和解释
- 禁止**顺手**做事情：不超出主线修任何其他问题，可以在发现后提出该问题，但不要一直辗转探索
- **同步**文档：修改代码后，如果修改的内容属于文档内描述的接口/需求/实现，则都需要同步更改

## 代码风格

- 函数式优先，组合优于继承，TS/JS 中避免 OOP
- 新功能优先复用/重构现有代码，不堆砌
- 优先复用项目内已有组件、样式模式和设计语言，避免为了局部需求重新发明一套交互或视觉规范
- KISS, DRY — 最简可行方案
- 重要：尽量不Fallback，找出真正的根因是什么，多去暴露错误，而不是掩盖错误
- 前端页面文案展示用户真正关心的功能信息、状态信息和操作提示，不展示面向开发过程的说明

## 架构与设计

- 从第一性原理解构问题 — 先明确什么是必须的，再决定怎么做
- 警惕 XY 问题 — 多角度审视方案，先确认真正要解决的是什么，主动提出替代方案
- 解决根本问题，不要 workaround — 如果现有架构不支持，重构它
- 质疑不合理的需求和方向 — 发现问题立刻指出，不要等我问才说，不要奉承或无脑赞同
- 架构设计时参考 ddia-principles 规则
- 技术选型推荐业内最佳实践 — 不确定时先 research，不要给过时的信息

# 关于subagent的通用规范

- **主 agent 只监督，不插手，派发后等待，仅 subagent 报错或偏离目标时介入**
   - **Why**：打断 subagent 的推理链就失去了使用subagent的意义，且主 agent 同时处理多任务会串上下文
- 交代两层目标
   - 整体目标：让 subagent 知道自己的产出如何融入大局，避免局部最优
   - 子任务目标：明确交付物，减少理解偏差
   - **Why**：subagent 没有全局视角，不知道"为什么做"，容易跑偏或过度设计
- 给足subagent上下文，比如：前置结果、业务背景、约束条件、参考资料
   - **Why**：subagent 不继承主 agent 的记忆，上下文不足会重复提问或假设错误
- 新任务必须新建 subagent，禁止复用
   - **Why**：复用会导致上下文容器累积，旧任务信息污染新任务，且模型有上下文长度限制，后期会遗忘关键约束


## 提交与 PR 规范

- Git 历史大体遵循 Conventional Commits，例如 `feat:`、`refactor:`、`style:`；推荐格式：`<type>(optional-scope): short summary`
- 保持提交聚焦且可运行；除非强相关，不要把前端和后端的大改混在同一个提交里
- PR 应包含：
  - 改了什么，以及为什么改
  - 关联的 issue 或任务（如果有）
  - API 或 UI 影响说明（有可见 UI 变化时附截图）
  - 本地验证步骤，例如 `go test ./...`、`pnpm build`
